🔐 GitHub access flow · admin vs user vs automatic

🛡️ Admin does this

  • One-time bootstrap: register Google OAuth, generate LIBRARY_VAULT_MASTER_KEY + LIBRARY_SESSION_SECRET, mint GITHUB_ADMIN_TOKEN (repo + admin:org), write all four into .env.prod on Hetzner.
  • One-time GitHub seeding: create ColaberryIntern/workspace-template with .claude/skills/, .mcp.json, README, USER_PROFILE.md; flag as Template repo via gh repo edit --template.
  • Per-user onboarding click: from /admin/users/new, enter email + display name + access level; tick create_workspace_repo. One POST runs tenancy.provision_user + workspaces.provision_user_workspace + scope grants in one transaction.
  • Approver matrix edits: commit/deploy config/library_approvers.json. Single source of truth for github_sync.can_approve().
  • Tenant-domain mapping: maintain config/library_tenant_domains.json. Unmatched domains hit queue_for_admin_review.
  • Token rotation steward: respond to vault.days_until_expiry() amber/red signals. Quarterly rotation of GITHUB_ADMIN_TOKEN is on the admin calendar.
  • Offboarding click: revoke every vault entry, remove user as collaborator, archive workspace repo (do not delete · preserves audit).
  • Unmatched-domain queue: approve or reject 403 queued_for_review entries when a new domain logs in.

👤 User does this

  • One-time Google SSO login at https://advisor.colaberry.ai/auth/login. First successful login provisions the User row via auth_google.provision_or_lookup_user().
  • Accept the GitHub collaborator invite for ColaberryIntern/{slug}-workspace. Until accepted, repo is admin-only.
  • Self-serve token paste at /admin/users/{me}/scopes · paste Gmail app password, Basecamp PAT, HubSpot key, etc. Token encrypted into vault keyed by user_id + tool_name. Plaintext never leaves the form.
  • Set TTL on every token (default 90 days). When amber, re-paste the rotated token. Backend never auto-refreshes · the user is the only person who can hold the new secret.
  • Clone workspace repo locally, run Claude Code. MCP runtime reads .mcp.json placeholders and calls vault.read_secret() at startup; plaintext stays in-process.
  • Submit drafts via the contributor publish form. Drafts land in the per-tenant moderation queue. Contributors cannot push to ColaberryIntern/AI_ProjectArchitect directly.
  • Approve in their lanes (Ram/Karun for sales categories, Kes for tech). Approval triggers github_pr_sync via the sync identity, not the approver's personal token.
  • Log out / report compromise: /auth/logout drops the library_session cookie. Suspected compromise = request admin to revoke vault entries.

⚙️ Provisioning flow · 10 steps

  1. adminOpens /admin/users/new, enters email + display name + access level (admin|approver|contributor|consumer); ticks create_workspace_repo.
  2. systemValidates domain against library_tenant_domains.json. colaberry.com matches → company_id=colaberry. Unmatched domains return 422 with queue_for_admin_review.
  3. systemtenancy.provision_or_lookup_user() creates the User row. Idempotent on email · re-running returns the existing user.
  4. systemIf access_level != consumer, workspaces.provision_user_workspace runs: repo_exists check, generate from template (or POST /orgs/.../repos), PUT collaborator with write, commit rendered USER_PROFILE.md + .mcp.json.
  5. systemScope grants: library:read for all; library:approve:{categories} from approvers matrix for approvers; platform:admin for admins. Audit row per grant.
  6. systemWelcome email via Mandrill: GitHub invite link, SSO login link, scopes link to paste tokens.
  7. userClicks Google SSO link, lands at /auth/callback. auth_google.exchange_code_for_userinfo() matches existing User row. JWT signed with LIBRARY_SESSION_SECRET, set as library_session httpOnly cookie 24h TTL.
  8. userAccepts GitHub collaborator invite. System has no callback · pure GitHub-side handshake.
  9. userNavigates to /admin/users/{me}/scopes, pastes tool tokens with TTL. vault.store_secret encrypts; plaintext wiped from the form on success.
  10. systemFirst Claude Code run reads .mcp.json placeholders, calls vault.read_secret(), injects decrypted values into the spawned MCP server's env. One decrypt audit row per server per session.

BoundaryHow it holds
🚧 Approver category isolationgithub_sync.can_approve(ram, 'skills') returns False. Cross-lane approval blocked at sync layer, not UI · the UI can lie, the matrix cannot.
🔒 No plaintext token over HTTPvault.store_secret takes plaintext only via form POST, immediately encrypts. read_secret returns plaintext only to backend caller_ids (mcp-runtime, gmail-fetcher-svc). Every decrypt audit-logged.
🏢 Tenant scoping on readsPer-tenant approval filter intersects every list query with the company_id in the session JWT. Bypass requires forging a JWT = needs LIBRARY_SESSION_SECRET = admin-only.

👤 User-scoped vault writesvault.store_secret enforces caller_id == user_id for non-admin callers. Cross-user writes are admin-only and audit-logged with caller_id != target_user_id.
🚫 Unmatched-domain rejectionauth_google does NOT auto-create a tenant for unknown domains. Returns 403 queued_for_review. Fail-closed.
🔄 Session secret rotation = kill switchRotating LIBRARY_SESSION_SECRET invalidates every JWT in one shot. For lost-laptop scenarios only; do not rotate on routine offboarding.
📨 Invite ≠ access until acceptedSystem creates repo + issues invite; user accepts on github.com. Until acceptance, repo is admin-only.

👆 One-click flow

  1. openFrom /my-day/ or /harvest/, user clicks a completed todo card. The row expands inline (no page change) into a Transformation Drawer on the right edge.
  2. rankDrawer shows 1-3 ranked suggestion chips at the top: 'agent · 92% match', 'prompt · 71% match', 'workflow · 58% match'. Each chip displays the scan signals that fired.
  3. pickUser clicks one chip (say 'agent'). Drawer body morphs into a pre-filled Submission form mapped to AssetMetadata fields for that category.
  4. citeEach field shows an 'extracted from' citation underneath ('from todo title', 'from comment #3 by Karun') so the user trusts the auto-fill. Fields are editable in place.
  5. submitUser edits anything wrong (usually nothing) and clicks 'Submit to library'. Calls the same store.submit flow as Add-to-Library; tags submission with source_task_bc_id so provenance is preserved.
  6. confirmDrawer collapses with a green 'Submitted · queued for review' toast and a link to the pending row. Todo card now shows a 'library: pending agent' badge so it's never proposed twice.
  7. reviewReviewer experience unchanged · submission appears in /library/pending queue. On accept, AssetMetadata.source_task links back to the originating BC todo, closing the loop both directions.

📄 Page design + proposed URL

/harvest/ · sibling of /my-day/ in the top nav, between My Day and Library.
"The new page lives at /harvest/ · a sibling of /my-day/ in the top nav, between 'My Day' and 'Library'. The name is deliberate: this is where finished work gets harvested into reusable assets. When you land on it, the first thing you see is a single sentence at the top: 'You completed 47 tasks in the last 30 days. 23 of them look transformable.' Below that, a horizontal strip of ten chips · one per asset kind · each showing the count of candidate transformations of that kind. Clicking a chip filters the list. The main body is a feed of completed-todo cards from output/ops/{user_id}/todos.json where status='completed', sorted by 'highest-confidence suggestion first'. Each card shows the todo title, the project, completed-at timestamp, and a colored ribbon along the left edge whose color matches the top suggested kind. On the right side of every card sits a stack of 1-3 suggestion chips with confidence scores. Clicking the card expands a right-edge Transformation Drawer. Already-transformed todos collapse to a one-line 'shipped to library' row at the bottom of the feed. The whole page reuses the library shell's left nav, so the user always sees library counts updating in real time as they submit · the dopamine loop is the whole point. No bulk-select, no wizards, no modals; the entire UX is: see card, click chip, glance at extracted fields, click Submit."
